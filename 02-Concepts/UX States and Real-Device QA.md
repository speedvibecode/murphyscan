# UX States and Real-Device QA

## Source Reels

- [[reel-08]] - every component needs loading, error, empty, and success states.
- [[reel-10]] - users will test edge cases if the builder does not.

## Why It Matters
Users rarely stay inside the ideal path. Blank screens, silent failures, unclear retries, and confusing first-use flows are product failures even when the backend works.

## Verification Questions
- Can a first-time user complete core flows without explanation?
- What happens on blank inputs, long inputs, repeated submits, and network failures?
- Does every core view expose loading, error, empty, and success states?
- Is there evidence of mobile and slow-network testing?

## Agent Notes
Treat missing error and empty states as product-readiness gaps, not polish.
